PROCÉDURE INTERNE · v1.1

Standardisation du démarrage de projet

Procédure officielle pour initier, documenter et versionner tout projet ERP ou application web chez Lithium Marketing. La quasi-totalité des opérations techniques est déléguée à Claude Code — le développeur se concentre sur les décisions, pas sur les commandes.

🖥️ Environnement : Windows / PowerShell ⏱️ Durée : ~5 minutes 🤖 Approche : Déléguée à Claude Code

Introduction

Ce document décrit la procédure standard que tout développeur doit suivre au démarrage d'un projet chez Lithium Marketing. L'objectif est d'uniformiser les pratiques pour que n'importe quel membre de l'équipe puisse reprendre, comprendre et maintenir un projet sans friction.

La procédure repose sur un principe simple : Claude Code exécute les commandes techniques (init Git, création de fichiers, remote, push, etc.) à votre place. Le développeur se contente de préparer l'environnement, de donner le bon prompt, et de valider les actions.

ℹ️ À propos de CLAUDE.md
Le fichier CLAUDE.md à la racine d'un projet est automatiquement chargé par Claude Code à chaque session. Il contient les règles, conventions et contexte spécifiques au projet, ce qui élimine le besoin de les répéter à chaque interaction.
🤖 Ce que fait Claude Code pour vous
  • Analyse du projet existant et détection de la stack
  • Génération du .gitignore adapté
  • Initialisation Git (git init)
  • Création de CLAUDE.md et README.md
  • Création et configuration du .env.example
  • Ajout du remote Lithium Marketing
  • Premier commit et push vers le remote
  • Tous les commits suivants, détaillés automatiquement

Prérequis

Avant de commencer, assurez-vous d'avoir sur votre machine :

  • Node.js (version 18 ou supérieure) — requis par Claude Code
  • Git installé et configuré (git config --global user.name et user.email)
  • Un compte Anthropic avec accès à Claude Code
  • Accès à l'organisation Lithium Marketing sur GitHub (ou la plateforme Git utilisée)
  • PowerShell 7+ recommandé (plutôt que Windows PowerShell 5.1)

Vérifier les versions installées

PowerShell 
# Vérifier Node.js et npm
node --version
npm --version

# Vérifier Git
git --version

# Vérifier la configuration Git
git config --global user.name
git config --global user.email

1Installation de Claude Code

Claude Code s'installe globalement via npm. Une seule installation par machine suffit ; il sera ensuite disponible depuis n'importe quel répertoire.

PowerShell 
# Installation globale
npm install -g @anthropic-ai/claude-code

# Vérifier l'installation
claude --version

Première authentification

Au premier lancement, Claude Code vous demandera de vous authentifier avec votre compte Anthropic.

PowerShell 
# Lancer Claude Code (ouvrira la fenêtre d'authentification)
claude
⚠️ Politique d'exécution PowerShell
Si PowerShell bloque l'exécution de scripts npm, lancez une fois en administrateur : Set-ExecutionPolicy -Scope CurrentUser RemoteSigned

2Préparation du projet

Cette étape reste manuelle car elle dépend de votre organisation locale et de la source des fichiers. L'objectif : avoir un dossier prêt à être ouvert par Claude Code.

Cas A — Nouveau projet (depuis zéro)

PowerShell 
# Se placer dans le dossier des projets Lithium
cd C:\Lithium\Projets

# Créer le dossier du nouveau projet
New-Item -ItemType Directory -Name "nom-du-projet"
cd nom-du-projet

Cas B — Projet existant (transfert de fichiers)

PowerShell 
# Créer le dossier et y copier les fichiers
cd C:\Lithium\Projets
New-Item -ItemType Directory -Name "nom-du-projet"
cd nom-du-projet

# Extraire l'archive dans le dossier courant
Expand-Archive -Path "C:\Chemin\Vers\archive.zip" -DestinationPath .

# OU copier depuis un dossier source
Copy-Item -Path "C:\Chemin\Source\*" -Destination . -Recurse
📁 Convention de nommage
Utilisez le kebab-case pour les noms de projets : erp-client-acme, site-vitrine-xyz. Évitez les espaces, majuscules et accents. Le nom doit être identique entre le dossier local et le dépôt distant.

3Création du dépôt distant

Créez le dépôt distant vide dans l'organisation Lithium Marketing via l'interface web de GitHub (ou la plateforme utilisée). Cette étape doit être faite avant de lancer Claude Code pour que le remote soit prêt à recevoir le push.

Paramètres du dépôt

  • Organisation : lithium-marketing
  • Nom : identique au dossier local (kebab-case)
  • Visibilité : Private par défaut
  • Ne pas cocher « Initialize with README »
  • Ne pas ajouter de .gitignore ni de licence
⚠️ Dépôt strictement vide
Si le dépôt distant contient déjà des fichiers (README, licence), le premier push de Claude échouera. Créez toujours un dépôt totalement vide.

Récupérer l'URL du dépôt

Copiez l'URL HTTPS (ou SSH) du dépôt nouvellement créé. Vous la fournirez à Claude à l'étape suivante.

Format attendu 
https://github.com/lithium-marketing/nom-du-projet.git
🔑 Authentification Git
Assurez-vous d'être authentifié à GitHub sur votre machine avant de demander à Claude de pusher. Utilisez gh auth login (GitHub CLI) ou un Personal Access Token configuré dans le credential manager Windows.

4Délégation à Claude Code

Lancez Claude Code dans le dossier du projet, puis donnez-lui le prompt standardisé ci-dessous. Claude s'occupe de tout le reste : analyse, création des fichiers, init Git, remote, commit et push.

PowerShell 
# Lancer Claude Code dans le dossier du projet
claude

Prompt standardisé Lithium Marketing

Copiez-collez ce prompt tel quel dans Claude Code. Remplacez uniquement l'URL du remote à la fin.

Prompt Claude Code — à copier tel quel 
Tu vas initialiser ce projet selon les standards Lithium Marketing.
Exécute les étapes suivantes dans l'ordre, en me demandant
confirmation avant toute action destructive :

1. ANALYSE
   Analyse le contenu du dossier et identifie la stack technique
   (langages, frameworks, outils). Liste ce que tu as trouvé.

2. FICHIERS DE BASE
   Crée (s'ils n'existent pas) :
   - .gitignore adapté à la stack détectée (secrets, node_modules,
     builds, fichiers d'IDE, fichiers OS)
   - .env.example listant les variables d'environnement attendues
     (sans valeurs réelles)
   - CLAUDE.md contenant :
       • Description technique du projet et stack
       • Structure des dossiers et leur rôle
       • Conventions de code à respecter
       • Commandes courantes (install, dev, build, test)
       • RÈGLE DE COMMIT : toujours faire des commits détaillés
         après chaque modification significative, avec un message
         structuré (type: sujet + description à puces des changements).
         Types autorisés : feat, fix, chore, docs, refactor, test, style.
       • Particularités du projet (intégrations, API, dépendances)
   - README.md technique destiné aux développeurs :
       • Présentation du projet
       • Prérequis et installation étape par étape
       • Commandes disponibles
       • Structure du projet
       • Variables d'environnement requises (sans valeurs)
       • Procédure de déploiement si applicable

3. GIT
   - Initialise Git avec la branche principale "main" (git init -b main)
   - Vérifie que .env, secrets et node_modules sont bien exclus
   - Fais le premier commit avec le message :
     "chore: initial project setup with Lithium Marketing standards"

4. REMOTE
   - Ajoute le remote origin : [COLLER ICI L'URL DU DÉPÔT LITHIUM]
   - Vérifie avec git remote -v
   - Pousse la branche main : git push -u origin main

5. RAPPORT FINAL
   Affiche un résumé : fichiers créés, stack détectée, URL du remote,
   et indique les prochaines étapes recommandées pour le développeur.

IMPORTANT :
- Demande-moi confirmation avant le push final
- Si un .env existe déjà dans le dossier, NE PAS le commiter et
  m'alerter immédiatement
- Si un dossier .git existe déjà, m'indiquer l'état avant d'agir
✅ Ce qui se passe automatiquement
Claude Code va analyser le projet, créer les fichiers, exécuter les commandes Git et pousser le code. Il vous demandera confirmation avant chaque action importante (premier commit, push). La règle de commits détaillés est inscrite dans CLAUDE.md et sera suivie à toutes les sessions futures.
⚠️ Surveillance active
Même si Claude exécute les commandes, restez attentif aux sorties de git status qu'il affichera. Si un fichier .env ou un secret apparaît dans la liste des fichiers à commiter, interrompez Claude immédiatement.

5Validation finale

Une fois Claude terminé, vérifiez que tout est en place avec ces quelques contrôles rapides.

PowerShell 
# Vérifier le statut Git (doit être clean)
git status

# Vérifier que le remote est bien configuré
git remote -v

# Vérifier que .env est bien ignoré
git check-ignore .env

# Consulter l'historique des commits
git log --oneline

Contrôles à effectuer dans l'interface web du dépôt

  • Le code est bien visible sur la plateforme
  • CLAUDE.md et README.md sont présents
  • Aucun .env n'est visible dans le dépôt
  • Aucun dossier node_modules ou équivalent
  • Le README s'affiche correctement sur la page d'accueil

Structure finale attendue

mon-projet/ ├── CLAUDE.md # Chargé automatiquement par Claude Code ├── README.md # Documentation développeur ├── .gitignore # Adapté à la stack ├── .env.example # Template des variables d'env ├── .env # Local uniquement, JAMAIS commité ├── src/ └── package.json

Commits suivants — Gérés automatiquement par Claude

Grâce à la règle inscrite dans CLAUDE.md, Claude Code créera des commits détaillés après chaque série de modifications, avec des messages suivant le format :

Format de commit standard 
type: sujet court à l'impératif

- Description du changement 1
- Description du changement 2
- Contexte ou motivation si pertinent

Types courants : feat, fix, chore, docs, refactor, test, style

🔐 Sécurité — Variables d'environnement et secrets

Aucune donnée sensible (clés API, mots de passe, tokens, chaînes de connexion) ne doit jamais être présente dans le code source ou dans l'historique Git.

Règle d'or : .env + .env.example

Chaque projet doit avoir ces deux fichiers :

Fichier Commité ? Rôle
.env ❌ Jamais Contient les vraies valeurs (secrets, clés, URL de base de données)
.env.example ✅ Oui Liste les variables attendues avec valeurs bidon ou vides. Sert de template.

Exemple de .env.example

.env.example 
# Base de données
DATABASE_URL=postgresql://user:password@localhost:5432/dbname

# API externes
STRIPE_API_KEY=sk_test_xxxxxxxxxxxxx
SENDGRID_API_KEY=SG.xxxxxxxxxxxxx

# Authentification
JWT_SECRET=change-me-in-production
SESSION_SECRET=change-me-in-production

# Environnement
NODE_ENV=development
PORT=3000

Que faire si un secret a été commité par erreur ?

  1. Révoquer immédiatement le secret auprès du service émetteur (Stripe, AWS, etc.)
  2. Générer un nouveau secret et le stocker dans .env
  3. Signaler l'incident au lead technique
  4. Si le dépôt est public ou partagé : nettoyer l'historique avec git filter-repo (contacter le lead)
🚨 Ne jamais faire
  • Commiter un .env « juste pour tester »
  • Coller un secret dans un message de commit
  • Partager un .env par Slack, email ou Teams — utilisez un gestionnaire de secrets
  • Push-force sur main sans autorisation du lead

✅ Checklist finale

Cliquez sur chaque case au fur et à mesure de votre progression. La checklist est persistante dans votre navigateur pour cette session.

  • Node.js, npm et Git installés et vérifiés
  • Claude Code installé globalement et authentifié
  • Authentification Git configurée (GitHub CLI ou PAT)
  • Dossier du projet créé avec un nom en kebab-case
  • Fichiers du projet récupérés (si applicable)
  • Dépôt distant créé vide dans l'organisation Lithium Marketing
  • URL du dépôt distant copiée
  • Claude Code lancé dans le dossier du projet (claude)
  • Prompt standardisé Lithium collé avec l'URL du remote
  • Claude a analysé le projet et créé .gitignore, .env.example, CLAUDE.md, README.md
  • Aucun secret détecté avant le commit
  • Git initialisé, premier commit effectué, remote ajouté et push réussi (par Claude)
  • Vérification finale : git status clean, git remote -v OK
  • Code visible sur la plateforme distante ✓
  • README s'affiche correctement sur la page d'accueil du dépôt

🛠️ Dépannage courant

claude: command not found

Le dossier global npm n'est pas dans le PATH. Ajoutez-le ou relancez PowerShell. Vérifiez avec npm config get prefix.

fatal: remote origin already exists

Un remote est déjà configuré. Supprimez-le avec git remote remove origin puis ajoutez-le à nouveau.

error: failed to push some refs

Le remote contient déjà des commits (README initial créé par erreur). Faites git pull origin main --rebase puis git push.

Authentification refusée sur git push

Utilisez un Personal Access Token à la place du mot de passe, ou configurez GitHub CLI avec gh auth login.

Fichier .env commité par erreur

Voir la section Sécurité. Révoquez les secrets immédiatement.
